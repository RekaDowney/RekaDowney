说明

　　本文旨在简单介绍Protostff的（反）序列化语法，同时介绍自己对Protostuff的（反）序列化过程的通用封装。Protostuff是一款高效的Java（反）序列化处理工具。网络上有不少关于Java各种（反）序列化工具的比较，大部分都提到Protostuff表现不错，那么就让我们来试试Protostuff到底该怎么使用吧！

（反）序列化工具Protostuff

　　Protostuff支持通过proto文件编译生成对应的Java（比较麻烦），也支持通过运行期动态获取类的Schema来执行序列化操作。本文主要介绍通过运行期方式执行序列化操作，proto编译操作不做介绍。

测试结果

　　序列化时，需要通过RuntimeSchema获取类的Schema，随后借助LinkedBuffer创建序列化缓存区，接着通过ProtostuffIOUtil#toByteArray将对象序列化成字节数组，最后释放先前创建的缓存区。

　　反序列化时，先要获取得到反序列化类的实例，随后通过ProtostuffIOUtil#mergeFrom将字节数组转换成实例的属性。在这里，由于限定了反序列化类为User，因此可以简单通过new User()方法而不是Objenesis来创建实例。

　　测试发现，对于普通对象，序列化效果看起来不错。但是，如果将上述的方法提炼成一个泛型方法来接收任意对象的（反）序列化（此时只能通过Objenesis来执行类的实例化），就会发现：Protostuff并不支持纯粹的Collection或者Map的反序列化。

　　Protostuff不支持反序列化Collection和Map，主要缺陷有：

  1. RuntimeSchema#getSchema方法不支持获取抽象类、接口、数组的Schema
  2. Objenesis无法实例化抽象类、接口、数组
  3. 即使在RuntimeSchema#getSchema和Objenesis实例化时限定了类为Collection和Map的实例（暂不考虑数组），在执行ProtostuffIOUtil#mergeFrom后得到的实例依然无法访问容器的成员。

　　此时的解决方案有两种，先介绍第一种方式。

使用protostuff-collectionschema（反）序列化Collection和Map

测试结果

　　上述针对List的（反）序列化操作与针对User的（反）序列化操作区别在于：通过protostuff-collectionschema提供的MessageCollectionSchema指定Collection的成员类型，相比之前的大幅度缩减序列化数组长度，本次序列化优势明显减弱。

　　由于是官方提供的Collection和Map的（反）序列化工具包，其性能可能好点，但是依然存在问题：如果容器嵌套会怎样？毫无疑问：针对List<Set<String>>、Map<String,List<String>>，List<Map<String,Set<String>>>等嵌套容器，protostuff-collectionschema又无法正常使用了。而且依然无法（反）序列化数组。

自定义（反）序列化工具类——Protostuffs

　　为解决protostuff无法很好地处理上述提及地各种集合、数组、接口、抽象类、枚举等地（反）序列化问题，包装器模式应运而生（好吧！这模式是瞎扯的）。包装器模式——“人”如其名，主要是通过借助一层外套将这些无法直接操作的数据类型包装成一个新的简单对象。该对象能够被Protostuff正常（反）序列化。

　　同时也将Protostuff（反）序列化过程抽取出来形成通用工具Protostuffs，并借助策略模式实现可定制的包装策略。
